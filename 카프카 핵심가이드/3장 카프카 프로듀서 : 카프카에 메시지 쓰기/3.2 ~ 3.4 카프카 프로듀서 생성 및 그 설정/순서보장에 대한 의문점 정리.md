- max.in.flight.requests.per.connection=1의 의미
    - 한 번에 한 개씩만 메시지 보냄
    ```A 보냄 → ACK 성공 → B 보냄 → ACK 성공```
    - 만약에 2라면 동시에 2개 요청 가능 따라서
      - 순서는 A -> B이지만 B가 먼저 ACK 성공을 받을 수 있어 순서가 꼬인다.
    - 나는 한 번에 하나씩 응답을 받고 다음 요청을 보내서 그러면 비동기가 아니라고 생각했는데
    - 코드 상에서는 비동기이다. 즉, send 자체는 비동기로 호출할 수 있으며(기본) 그렇지만 요청을 보낼 때 내부 버퍼가 있어서 순서대로 한번에 하나의 요청이 가도록 보장하는것이다.
    - 번외로 in-flight란 프로듀서가 브로커에 메시지를 보내고 아직 응답을 받지 못한 상태의 요청이다.
- enable.idempotence = false의 의미 
  - 시나리오
    -  프로듀서가 메시지 A를 브로커에 보냄 
    - 브로커는 정상적으로 메시지를 저장함 
    - 그런데 ACK 응답이 네트워크 문제로 프로듀서에 도달하지 않음 
    - 프로듀서는 “ACK 못 받았네? 그럼 실패로 간주하고 재전송!” 
    - 브로커 입장에선 → 똑같은 메시지 A가 두 번 옴 
    - 별다른 처리 없으면 A를 두 번 저장하게 됨 → 중복 발생!
  - 따라서 true라면 producer가 레코드에 시퀀스 번호를 넣기 때문에 이 시퀀스 번호를 가지고
  - 브로커가 중복된 메시지인지 판단한다.
